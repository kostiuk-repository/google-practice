# Medium_05_StringIntegerAtoi — Довідка

## Техніка: Manual Character Scan + Overflow Guard

**Суть:** рядок читається вручну зліва направо через явний покроковий стан
парсера — індекс `i`, знак `sign`, накопичене значення `result`. Спочатку
пропускаються пробіли, потім опційно зчитується знак `+`/`-`, потім у циклі
зчитуються цифри, і кожна нова цифра "домножує й додає" `result`. Ключовий
нюанс — перевірка переповнення робиться ДО множення/додавання (порівнянням
з `Integer.MAX_VALUE / 10` та `Integer.MAX_VALUE % 10`), бо після
переповнення `int` значення вже спотворене і перевіряти post-factum пізно.

```java
int i = 0, sign = 1;
long result = 0; // або int з ручною overflow-guard перевіркою
while (i < s.length() && Character.isDigit(s.charAt(i))) {
    int digit = s.charAt(i) - '0';
    if (result > (Integer.MAX_VALUE - digit) / 10) {
        return sign == 1 ? Integer.MAX_VALUE : Integer.MIN_VALUE;
    }
    result = result * 10 + digit;
    i++;
}
```

**Коли застосовувати:** будь-який ручний парсинг числа з рядка, де не
можна просто покластися на `Integer.parseInt` (бо потрібен контроль над
переповненням, кастомна основа числення, часткове зчитування префіксу,
або валідація формату замість кидання винятку).

**Складність:** O(N) час (один прохід по рядку), O(1) додаткова пам'ять.

## Drills

1. **Medium_05_01** — чистий скан цифр без знаку і без overflow guard (найпростіший зріз циклу).
2. **Medium_05_02** — додається обробка знаку `+`/`-`, overflow guard все ще не потрібен.
3. **Medium_05_03** — той самий overflow guard, але перевиведений під межі `byte` замість `int`.
4. **Medium_05_04** — overflow guard використовується як предикат валідації всього рядка, а не clamp.
5. **Medium_05_05** — та сама покрокова акумуляція, але база 16 (hex) замість бази 10.

Розв'язуй у порядку 01 → 05, не підглядаючи в Medium_05_StringIntegerAtoi.java.
