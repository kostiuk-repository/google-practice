# Easy_10_FizzBuzz — Довідка

## Суть теми / алгоритму простими словами
Розв'язання задачі FizzBuzz зазвичай починається з простого ланцюжка розгалужень `if-else`. Однак при збільшенні кількості умов (наприклад, додаванні дільників 7, 11, 13 тощо) цей ланцюжок розростається експоненціально, оскільки доводиться вручну перебирати всі комбінації перетинів (наприклад, 3 і 5, 3 і 7, 5 і 7, 3 і 5 і 7).
Техніка **Extensible Rule List (Розширюваний список правил)** вирішує цю проблему за допомогою винесення логіки "дільник $\to$ слово" у структуру даних (наприклад, список або масив пар). Для кожного числа ми послідовно перевіряємо всі правила зі списку, дописуючи слово в буфер у разі подільності. Якщо буфер після перевірки всіх правил виявився порожнім, ми просто записуємо саме число.

Базова задача: згенерувати список рядків від 1 до $N$, замінюючи числа, що діляться на 3, на "Fizz", ті, що на 5 — на "Buzz", а ті, що на обидва дільники — на "FizzBuzz".

## Інженерний контекст: маленький rule engine

FizzBuzz перевіряє не `%`, а спосіб організувати **пересічні правила**. Незалежні правила можуть одночасно спрацювати, а output є композицією їхніх labels.

### Де зустрічається

- tagging events кількома незалежними labels;
- feature flags і policy evaluation;
- validation pipeline, що накопичує всі порушення;
- routing rules, alert classification і log enrichment;
- configurable formatters, де правило є парою `predicate → output`.

Для двох фіксованих правил найчитабельніший `if/else` цілком достатній. Rule list стає корисним, коли правила конфігуруються, їх багато або треба повертати всі збіги. Це trade-off між простотою конкретного рішення та extensibility.

### Що перевіряє співбесіда

- чи перевірите найбільш специфічну умову першою в `if/else`;
- або чи уникнете combinatorial branches через незалежне накопичення labels;
- чи не додасте число, якщо спрацювало хоча б одне правило;
- чи поясните complexity як $O(n \cdot r)$ для `r` правил;
- чи не over-engineer-ите базовий FizzBuzz складною hierarchy класів.

### a) Традиційне розширення правил для числа `i = 15`
Набір правил (впорядкованих за зростанням):
1. Ділиться на 3 $\to$ додати `"Fizz"`
2. Ділиться на 5 $\to$ додати `"Buzz"`

Початковий стан: `StringBuilder sb = new StringBuilder()`.

| Перевірка правила | Умова `15 % divisor == 0` | Дія над `StringBuilder` | Поточний вміст `sb` |
| :--- | :--- | :--- | :--- |
| **Правило 1 (3 -> "Fizz")** | `15 % 3 == 0` (Так) | `sb.append("Fizz")` | `"Fizz"` |
| **Правило 2 (5 -> "Buzz")** | `15 % 5 == 0` (Так) | `sb.append("Buzz")` | `"FizzBuzz"` |

Оскільки `sb.length() > 0`, результатом для числа 15 є `"FizzBuzz"`.

### b) Звичайний випадок, коли правила не спрацьовують: `i = 4`
| Перевірка правила | Умова `4 % divisor == 0` | Дія над `StringBuilder` | Поточний вміст `sb` |
| :--- | :--- | :--- | :--- |
| **Правило 1 (3 -> "Fizz")** | `4 % 3 == 0` (Ні) | нічого | `""` |
| **Правило 2 (5 -> "Buzz")** | `4 % 5 == 0` (Ні) | нічого | `""` |

Оскільки `sb.length() == 0`, результатом для числа 4 є `"4"` (рядок представлення самого числа).

## Ключова ідея та шаблон коду (Core Concept & Code Template)

Використання `StringBuilder` та списку об'єктів-правил (наприклад, кастомного класу `Rule` або вбудованого `Map.Entry`) робить код Open-Closed: відкритим для розширення, але закритим для змін. Додавання нового правила не вимагає переписування жодного рядка умовної логіки в циклі, достатньо лише додати елемент до списку правил.

### Базовий шаблон коду (Java)
```java
import java.util.ArrayList;
import java.util.List;

public class Easy_10_FizzBuzz {

    // Допоміжний клас для представлення правила
    private static class Rule {
        int divisor;
        String word;
        
        Rule(int divisor, String word) {
            this.divisor = divisor;
            this.word = word;
        }
    }

    public List<String> fizzBuzz(int n) {
        List<String> result = new ArrayList<>();
        
        // Ініціалізація списку правил
        List<Rule> rules = new ArrayList<>();
        rules.add(new Rule(3, "Fizz"));
        rules.add(new Rule(5, "Buzz"));
        
        for (int i = 1; i <= n; i++) {
            StringBuilder sb = new StringBuilder();
            
            // Застосовуємо кожне правило до поточного числа
            for (Rule rule : rules) {
                if (i % rule.divisor == 0) {
                    sb.append(rule.word);
                }
            }
            
            // Якщо жодне правило не спрацювало, записуємо саме число
            if (sb.length() == 0) {
                result.add(String.valueOf(i));
            } else {
                result.add(sb.toString());
            }
        }
        
        return result;
    }
}
```

### Аналіз складності (Complexity Analysis)
- **Time Complexity (Часова складність):** $O(N \cdot R)$, де $N$ — верхня межа діапазону, а $R$ — кількість правил. Оскільки кількість правил зазвичай є малою константою (наприклад, 2 чи 4), часова складність є лінійною — $O(N)$.
- **Space Complexity (Просторова складність):** $O(R)$ додаткової пам'яті для зберігання списку правил. Для виходу програми створюється список довжини $N$, що вимагає $O(N)$ пам'яті.

## Граничні випадки та підводні камені (Edge Cases & Common Pitfalls)

1. **Неправильний порядок правил:** Якщо правила перевіряються в хаотичному порядку, то для числа 15 ми можемо отримати `"BuzzFizz"` замість `"FizzBuzz"`. Правила у списку мають бути строго впорядковані за логікою очікуваного виходу.
2. **Використання `String` замість `StringBuilder`:** Конкатенація рядків `String s = ""; s += "Fizz"` у циклі для кожного числа створює проміжні об'єкти в пам'яті. Використання `StringBuilder` є більш оптимальним.
3. **Неефективний `if-else` при масштабуванні:** При додаванні дільника `7` наївний підхід вимагає 8 гілок `if-else` (3, 5, 7, 3+5, 3+7, 5+7, 3+5+7, жодне). Це призводить до помилок "копіювати-вставити".
