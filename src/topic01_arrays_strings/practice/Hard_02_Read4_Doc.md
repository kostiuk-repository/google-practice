# Hard_02_Read4 — Довідка

## Техніка: Stateful Chunked-Reader Wrapping (Leftover Buffer)

**Суть:** низькорівневий `read4` завжди повертає рівно до 4 символів за
виклик — незалежно від того, скільки символів насправді потрібно
поточному виклику `read`/`readLine`/тощо. Тому обгортка тримає на рівні
екземпляра невеликий буфер (`internalBuffer`), вказівник на першу
непрочитану позицію в ньому (`internalPtr`) та кількість дійсних символів
у буфері (`internalCount`). Кожен публічний виклик спочатку "доїдає"
залишки з `internalBuffer`, і лише коли буфер порожній — викликає `read4`
знову, зберігаючи надлишок символів для наступного виклику.

```java
private char[] internalBuffer = new char[4];
private int internalPtr = 0;
private int internalCount = 0;

private char nextChar() {
    if (internalPtr == internalCount) {
        internalCount = read4(internalBuffer);
        internalPtr = 0;
        if (internalCount == 0) return '\0'; // EOF
    }
    return internalBuffer[internalPtr++];
}
```

**Коли застосовувати:** будь-яка обгортка над "чанкованим" (chunked)
низькорівневим API, яке видає дані фіксованими блоками, коли
високорівневий метод повинен видавати дані довільними порціями і може
викликатися ПОВТОРНО — стан між викликами обов'язково повинен пережити
виклик методу, інакше "зайві" прочитані символи будуть втрачені.

**Складність:** O(N) час на N корисних символів (кожен символ файлу
проходить через `read4` рівно один раз), O(1) додаткова пам'ять
(буфер фіксованого розміру 4, не залежить від N).
