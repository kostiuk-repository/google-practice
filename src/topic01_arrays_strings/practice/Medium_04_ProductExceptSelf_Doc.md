# Medium_04_ProductExceptSelf — Довідка

## Суть теми / алгоритму простими словами
Задача "Добуток масиву, крім себе" (Product of Array Except Self) вимагає знайти для кожного елемента $nums[i]$ добуток усіх інших елементів масиву. 
Основні обмеження:
1. Часова складність має бути лінійною — $\mathcal{O}(N)$.
2. **Не можна використовувати операцію ділення** (що робить неефективним наївне обчислення загального добутку з подальшим діленням `totalProduct / nums[i]`).
3. (Бажано) додаткова пам'ять має бути $\mathcal{O}(1)$.

Для розв'язання використовують техніку **Префіксного та Суфіксного проходів (Prefix Sweep + Suffix Sweep)**. 
- **Префіксний масив** зберігає добуток усіх елементів, які стоять *ліворуч* від поточного елемента.
- **Суфіксний масив** зберігає добуток усіх елементів, які стоять *праворуч* від нього.
- Для кожної позиції $i$ відповідь дорівнює: $\text{prefix}[i] \times \text{suffix}[i]$.

## Інженерний контекст: задача не про множення

На співбесіді **Product Except Self** виглядає як штучне обмеження: «не використовуй ділення, зроби за $O(n)$». Насправді вона тренує значно ширшу операцію:

> Для кожної позиції `i` обчислити агрегат усіх елементів, крім `i`, не повторюючи повний прохід і не покладаючись на обернену операцію.

Множення — лише один конкретний агрегат. Та сама схема працює для суми, `min`, `max`, `gcd`, bitwise `AND/OR`, об’єднання множин, композиції функцій або матричних перетворень — якщо операція асоціативна й має нейтральний елемент.

### 1. Алгебраїчна модель: monoid замість «масиву чисел»

Нехай є операція $\otimes$:

- **асоціативність:** $(a \otimes b) \otimes c = a \otimes (b \otimes c)$;
- **нейтральний елемент:** $e \otimes a = a \otimes e = a$.

Тоді:

$$
answer[i] =
(a_0 \otimes \dots \otimes a_{i-1})
\otimes
(a_{i+1} \otimes \dots \otimes a_{n-1})
$$

Префікс зберігає ліву частину, суфікс — праву. Обернена операція не потрібна.

| Агрегат | $\otimes$ | Нейтральний елемент | Чи завжди є inverse? |
|---|---|---|---|
| добуток | `×` | `1` | ні: нуль не має мультиплікативного inverse |
| сума | `+` | `0` | для чисел так, але не для всіх типів |
| мінімум | `min` | `+∞` | ні |
| максимум | `max` | `-∞` | ні |
| bit flags | `OR` | `0` | ні |
| множини | union | порожня множина | ні |
| матриці | matrix multiply | identity matrix | лише для invertible matrices |

> Важлива межа: для некомутативної операції, наприклад матричного множення, порядок елементів не можна змінювати. Асоціативність дозволяє перегрупування, але не перестановку.

### 2. Паралельні та розподілені обчислення

Prefix scan — стандартна collective-операція, а не лише інтерв’ю-патерн. Наприклад, [`MPI_Exscan`](https://www.mpi-forum.org/docs/mpi-2.2/mpi22-report/node116.htm) виконує exclusive prefix reduction над даними, розподіленими між процесами. У distributed-версії leave-one-out aggregation можна:

1. обчислити exclusive prefix зліва;
2. обчислити exclusive suffix у зворотному порядку;
3. скомбінувати локальні результати для кожного rank/partition.

Apache Spark має деревоподібні агрегації на зразок [`RDD.treeAggregate`](https://spark.apache.org/docs/latest/api/python/reference/api/pyspark.RDD.treeAggregate.html), де локальні результати комбінуються асоціативною функцією. Це не готова реалізація Product Except Self, але та сама вимога асоціативності дозволяє ефективно будувати багаторівневі scans і reductions.

Практичні сценарії:

- агрегат станів усіх shards, крім поточного;
- конфігурація або feature flags усіх сусідніх вузлів;
- leave-one-partition-out метрики;
- паралельні cumulative transformations;
- підготовка даних до tree reduction або scan на GPU/кластері.

### 3. Бази даних і аналітичні вікна

SQL описує схожу потребу декларативно: «агрегуй frame, але виключи поточний рядок». PostgreSQL підтримує window frame option [`EXCLUDE CURRENT ROW`](https://www.postgresql.org/docs/current/sql-expressions.html).

```sql
SELECT
    id,
    SUM(value) OVER (
        ORDER BY id
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
        EXCLUDE CURRENT ROW
    ) AS sum_except_self
FROM metrics;
```

Це семантичний аналог задачі. Конкретний query plan визначає оптимізатор БД, тому не слід стверджувати, що будь-яка СУБД буквально створює два Java-подібні масиви prefix/suffix.

### 4. Геометрія, графіка та ML

Замінимо добуток на агрегат bounding box:

```text
combine(boxA, boxB) =
    [min(x1), min(y1), max(x2), max(y2)]
```

Для кожного об’єкта можна побудувати bounding box усіх **інших** об’єктів через prefix/suffix boxes. Це корисна building block для leave-one-out геометричних перевірок, spatial partitioning та аналізу впливу одного detection.

У графіці аналогічно комбінуються ordered transformations. Якщо треба швидко відповісти «яка композиція всіх трансформацій, крім вузла `i`», лівий і правий cumulative products прибирають повторний $O(n)$ перерахунок для кожного `i`.

### 5. Надійність: чому inverse — крихка оптимізація

Підхід `total / nums[i]` працює лише якщо:

- операція має inverse;
- поточний елемент не є нулем;
- арифметика не втратила точність;
- переповнення total не сталося раніше;
- тип результату підтримує потрібну операцію.

Prefix/suffix формулює алгоритм через **пряму операцію**, тому нулі обробляються природно:

- один нуль → ненульовим може бути лише результат на позиції нуля;
- два нулі → всі результати нульові;
- жодних окремих `if` для цього не потрібно.

Це не означає, що prefix/suffix завжди єдиний варіант. Для invertible операції можна використати total + inverse; для статичних range queries — segment tree або sparse table; для потокових update/query — Fenwick/segment tree; для distributed data — parallel scan. Вибір залежить від контракту.

## Що саме перевіряє співбесіда

### Рівень 1 — побачити квадратичний повтор

Кандидат має помітити, що наївне рішення повторно множить майже ті самі елементи:

```text
i=0:       2×3×4
i=1:     1×  3×4
i=2:     1×2×  4
i=3:     1×2×3
```

### Рівень 2 — розділити інформацію на дві незалежні частини

Відповідь для `i` складається з:

- агрегату строго ліворуч;
- агрегату строго праворуч.

Це основний conceptual jump.

### Рівень 3 — правильно використати нейтральний елемент

Для крайньої позиції одна сторона порожня. Порожній добуток дорівнює `1`, тому формула не потребує спеціальної гілки.

### Рівень 4 — стиснути пам’ять

Спочатку чесно побудуйте `prefix[]` і `suffix[]`. Потім:

- збережіть prefix одразу в `answer[]`;
- suffix тримайте в одній змінній;
- рухайтеся справа наліво.

Так auxiliary space зменшується з $O(n)$ до $O(1)$ без погіршення читабельності інваріанта.

### Рівень 5 — пояснити trade-offs

Сильна відповідь на співбесіді включає:

- чому два послідовні проходи все ще $O(n)$;
- чому `answer[]` зазвичай не рахується як auxiliary memory;
- як поводяться нулі;
- де можливе `int` overflow і чи потрібен `long`;
- чому операція має бути асоціативною;
- чому для некомутативної операції важливий порядок.

На початку другого проходу `answer[i]` уже містить лише ліву частину, а `suffix` — лише праву; тому множення ніколи не включає `nums[i]`.

### a) Звичайний випадок: масив `nums = [1, 2, 3, 4]`

#### Крок 1: Префіксний прохід (зліва направо)
Накопичуємо добуток елементів *до* індексу $i$. Для першого елемента префікс за замовчуванням дорівнює $1$ (нейтральний елемент для множення).
- `prefix[0] = 1`
- `prefix[1] = prefix[0] * nums[0] = 1 * 1 = 1`
- `prefix[2] = prefix[1] * nums[1] = 1 * 2 = 2`
- `prefix[3] = prefix[2] * nums[2] = 2 * 3 = 6`

Масив префіксів: `[1, 1, 2, 6]`

#### Крок 2: Суфіксний прохід (справа наліво)
Замість створення окремого масиву суфіксів ми будемо підтримувати одну змінну `suffix`, яка спочатку дорівнює $1$. Ми проходимо по масиву `prefix` з кінця до початку і множимо його значення на поточний `suffix`.

| Індекс $i$ | Початковий `prefix[i]` | Поточний `suffix` | Обчислене значення `ans[i]` | Оновлення `suffix` (`suffix *= nums[i]`) |
| :--- | :--- | :--- | :--- | :--- |
| **$i = 3$** | `6` | `1` | `6 * 1 = 6` | `1 * nums[3] = 1 * 4 = 4` |
| **$i = 2$** | `2` | `4` | `2 * 4 = 8` | `4 * nums[2] = 4 * 3 = 12` |
| **$i = 1$** | `1` | `12` | `1 * 12 = 12` | `12 * nums[1] = 12 * 2 = 24` |
| **$i = 0$** | `1` | `24` | `1 * 24 = 24` | `24 * nums[0] = 24 * 1 = 24` |

**Фінальний результат:** `[24, 12, 8, 6]`.

---

### b) Граничний випадок: наявність нуля в масиві `nums = [1, 0, 3, 4]`

#### Крок 1: Префіксний прохід
- `prefix[0] = 1`
- `prefix[1] = prefix[0] * nums[0] = 1 * 1 = 1`
- `prefix[2] = prefix[1] * nums[1] = 1 * 0 = 0`
- `prefix[3] = prefix[2] * nums[2] = 0 * 3 = 0`

Масив префіксів: `[1, 1, 0, 0]`

#### Крок 2: Суфіксний прохід (починаючи з `suffix = 1`)
- **$i = 3$**: `ans[3] = prefix[3] * suffix = 0 * 1 = 0`. Оновлення `suffix = 1 * 4 = 4`.
- **$i = 2$**: `ans[2] = prefix[2] * suffix = 0 * 4 = 0`. Оновлення `suffix = 4 * 3 = 12`.
- **$i = 1$**: `ans[1] = prefix[1] * suffix = 1 * 12 = 12`. Оновлення `suffix = 12 * 0 = 0`.
- **$i = 0$**: `ans[0] = prefix[0] * suffix = 1 * 0 = 0`. Оновлення `suffix = 0 * 1 = 0`.

**Фінальний результат:** `[0, 12, 0, 0]`. Алгоритм спрацював ідеально без винятків типу `ArithmeticException`.

## Ключова ідея та шаблон коду

Оскільки ділення заборонене, ми розкладаємо добуток елементів на дві незалежні частини:
$$ \prod_{j \neq i} nums[j] = \left( \prod_{j=0}^{i-1} nums[j] \right) \times \left( \prod_{j=i+1}^{n-1} nums[j] \right) $$
Перша дужка — це префіксний добуток, друга — суфіксний. Ми можемо порахувати їх за лінійний час. 
Для оптимізації по пам'яті ми спочатку записуємо префікси прямо у вихідний масив `result`, а потім під час зворотного проходу модифікуємо його з використанням змінної накопичувального суфікса `suffix`.

### Java-шаблон (Оптимізований по пам'яті)
Доступ до файлу: [Medium_04_ProductExceptSelf.java](file:///C:/Users/kosti/Projects/google-practice/src/topic01_arrays_strings/practice/Medium_04_ProductExceptSelf.java)

```java
public int[] productExceptSelf(int[] nums) {
    int n = nums.length;
    int[] result = new int[n];
    
    // Крок 1: Обчислюємо префіксний добуток і пишемо його в result
    result[0] = 1; // перед першим елементом нічого немає
    for (int i = 1; i < n; i++) {
        result[i] = result[i - 1] * nums[i - 1];
    }
    
    // Крок 2: Зворотний суфіксний прохід із running-змінною
    int suffix = 1;
    for (int i = n - 1; i >= 0; i--) {
        result[i] = result[i] * suffix; // комбінуємо префікс і суфікс
        suffix = suffix * nums[i];      // оновлюємо суфікс для наступного кроку
    }
    
    return result;
}
```

### Аналіз складності:
- **Часова складність (Time Complexity):** $\mathcal{O}(N)$, де $N$ — розмір масиву. Ми робимо два послідовні проходи по масиву.
- **Просторова складність (Space Complexity):** $\mathcal{O}(1)$ додаткової пам'яті, оскільки масив результату не враховується в загальну оцінку складності за умовою задачі.

## Граничні випадки та підводні камені

1. **Чому ділення не працює в загальному випадку**:
   - Якщо в масиві є **один нуль**, загальний добуток буде `0`. При діленні `0 / nums[i]`, для ненульових елементів ми отримаємо `0` (що правильно), але для самого нуля ми отримаємо `0 / 0` (помилка).
   - Якщо в масиві є **два або більше нулів**, то для будь-якого елемента (включаючи самі нулі) добуток "крім себе" завжди дорівнює `0`. Загальний добуток буде `0`, але ми все одно не зможемо розділити `0 / 0`.
2. **Нейтральний елемент (Identity)**:
   - При множенні базовим початковим значенням має бути `1`. 
   - *Помилка:* Якщо ініціалізувати `1` як `0` (наприклад, помилково сплутавши з сумою), результатом роботи всього алгоритму будуть нулі.
3. **Off-by-one при заповненні префіксів**:
   - Зверніть увагу: `result[i]` використовує значення `nums[i - 1]`. Завжди перевіряйте, щоб індекс `nums[i - 1]` не вийшов за межі масиву при `i = 0`.

- **Підказка щодо модифікації:** 
  Нейтральним елементом для максимуму є мінімально можливе число: `Integer.MIN_VALUE`. 
  1. Створіть масив префіксних максимумів: `prefixMax[0] = Integer.MIN_VALUE`, `prefixMax[i] = Math.max(prefixMax[i - 1], nums[i - 1])`.
  2. Заведіть змінну `int suffixMax = Integer.MIN_VALUE`.
  3. Пройдіть з кінця та обчисліть: `result[i] = Math.max(prefixMax[i], suffixMax)` та оновіть `suffixMax = Math.max(suffixMax, nums[i])`.
- **Приклад:**
  - Вхід: `[3, 1, 4, 1, 5]`
  - Вихід: `[5, 5, 5, 5, 4]`.
