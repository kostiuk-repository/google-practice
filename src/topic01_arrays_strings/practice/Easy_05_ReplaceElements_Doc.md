# Easy_05_ReplaceElements — Довідка

## Суть теми / алгоритму простими словами
Техніка **Suffix/Prefix Running Aggregate (Накопичувальний агрегат суфікса чи префікса)** полягає в обході масиву з одного кінця до іншого при збереженні проміжного накопиченого значення (наприклад, максимуму чи мінімуму). 
Замість того, щоб для кожного елемента заново шукати екстремум серед усіх його сусідів праворуч (чи ліворуч), що вимагало б вкладеного циклу та давало складності $O(N^2)$, ми робимо один прохід і накопичуємо результат у змінній. Найважливіший нюанс: оскільки значення поточного елемента не повинно впливати на його власний результат (а лише на наступні кроки), ми спочатку записуємо поточний агрегат, а вже потім оновлюємо його самим елементом.

Базова задача: замінити кожен елемент масиву найбільшим елементом серед тих, що розташовані праворуч від нього, а останній елемент замінити на `-1`.

## Інженерний контекст: стиснення майбутнього в один стан

Replace Elements ставить незручне питання: відповідь для `i` залежить від елементів, яких forward scan ще не бачив. Замість зберігати «майбутнє», ми змінюємо напрямок обходу й стискаємо вже побачений suffix до одного `rightMax`.

### Де зустрічається

- precomputation максимуму/мінімуму на майбутньому часовому горизонті;
- right-to-left dynamic programming;
- suffix risk/limit/availability summaries;
- перевірка, чи домінує поточний елемент над усіма наступними;
- підготовка допоміжних масивів для range queries або greedy-рішень.

Узагальнення: якщо відповідь для `i` є асоціативним агрегатом `a[i+1..n-1]`, suffix scan часто замінює вкладений цикл. Для `max` нейтральне початкове значення має відповідати контракту; `-1` коректне тут лише через обмеження вхідних значень.

### Що перевіряє співбесіда

- чи побачите, що правильний напрямок прибирає потребу «знати майбутнє»;
- чи збережете оригінальний `arr[i]` до in-place перезапису;
- чи не включите поточний елемент у власний suffix;
- чи сформулюєте інваріант `rightMax`;
- чи відрізните строгий suffix `i+1..n-1` від inclusive suffix `i..n-1`.

## Ключова ідея та шаблон коду (Core Concept & Code Template)

Головна ідея — обходити масив **з кінця (справа наліво)**. Завдяки цьому ми завжди знаємо максимальне значення з правої частини масиву без повторних циклів. Для оновлення елементів in-place (без виділення нового масиву) ми використовуємо тимчасову змінну `temp` для збереження початкового значення `arr[i]` перед тим, як перезаписати його поточним максимумом `maxRight`.

### Базовий шаблон коду (Java)
```java
public int[] replaceElements(int[] arr) {
    // Ініціалізуємо максимальний елемент праворуч як -1 за умовою задачі
    int maxRight = -1;
    
    // Рухаємося від останнього елемента до першого
    for (int i = arr.length - 1; i >= 0; i--) {
        // Тимчасово зберігаємо оригінальне значення, оскільки ми його перезапишемо
        int temp = arr[i];
        
        // Перезаписуємо поточний елемент максимумом з правої частини
        arr[i] = maxRight;
        
        // Оновлюємо максимум праворуч з урахуванням оригінального збереженого елемента
        maxRight = Math.max(maxRight, temp);
    }
    
    return arr;
}
```

### Аналіз складності (Complexity Analysis)
- **Time Complexity (Часова складність):** $O(N)$, де $N$ — довжина масиву `arr`. Ми робимо один лінійний прохід по масиву.
- **Space Complexity (Просторова складність):** $O(1)$ додаткової пам'яті, оскільки всі зміни виконуються in-place без створення нових масивів.

## Граничні випадки та підводні камені (Edge Cases & Common Pitfalls)

1. **Несвоєчасне оновлення змінної `maxRight`:** Якщо оновити `maxRight = Math.max(maxRight, arr[i])` до того, як перезаписати `arr[i]`, поточний елемент буде перезаписаний максимумом, який містить його самого. Це порушує умову "елементи строго праворуч". Наприклад, для `[1, 5]` ви отримаєте `[5, 5]` замість `[5, -1]`.
2. **Забута тимчасова змінна:** Спроба записати `arr[i] = maxRight` і після цього `maxRight = Math.max(maxRight, arr[i])` призведе до втрати оригінального значення `arr[i]`, оскільки воно вже затерте. Завжди використовуйте `temp` для зберігання вихідного значення.
3. **Обхід зліва направо у наївній реалізації:** Спроба шукати максимум праворуч за допомогою вкладеного циклу призведе до перевищення ліміту часу (Time Limit Exceeded) на великих масивах через складність $O(N^2)$.

### 1. [Easy_05_01_ReplaceElements.java](file:///C:/Users/kosti/Projects/google-practice/src/topic01_arrays_strings/practice/Easy_05_01_ReplaceElements.java) — Replace with Smallest on Right
- **Мета:** Замінити кожен елемент на найменший елемент праворуч від нього. Останній елемент = `-1`.
- **Відмінність від основної задачі:** Замість знаходження максимуму шукається мінімум.
- **Підказка щодо модифікації:** Визначте мінімум праворуч. Для ініціалізації мінімуму для останнього кроку зручно почати з `minRight = -1`, але при оновленні перевіряти, чи це перша ітерація, або ініціалізувати першим припустимим значенням. Якщо всі числа додатні, можна використати `minRight = -1` як спеціальний маркер.
- **Приклад:**
  - Вхід: `arr = [17, 18, 5, 4, 6, 1]`
  - Вихід: `[1, 1, 1, 1, 1, -1]`.

### 2. [Easy_05_02_ReplaceElements.java](file:///C:/Users/kosti/Projects/google-practice/src/topic01_arrays_strings/practice/Easy_05_02_ReplaceElements.java) — Replace with Greatest on Left
- **Мета:** Замінити кожен елемент найбільшим значенням ліворуч від нього. Перший елемент має стати `-1`.
- **Відмінність від основної задачі:** Обхід здійснюється **зліва направо** (Prefix Running Aggregate).
- **Підказка щодо модифікації:** Ініціалізуйте `maxLeft = -1`. Рухайтеся від `i = 0` до `arr.length - 1`, використовуйте `temp = arr[i]`, записуйте `arr[i] = maxLeft` і оновлюйте `maxLeft = Math.max(maxLeft, temp)`.
- **Приклад:**
  - Вхід: `arr = [1, 10, 4, 20]`
  - Вихід: `[-1, 1, 10, 10]`.

### 3. [Easy_05_03_ReplaceElements.java](file:///C:/Users/kosti/Projects/google-practice/src/topic01_arrays_strings/practice/Easy_05_03_ReplaceElements.java) — Element Minus Greatest on Right
- **Мета:** Замінити кожен елемент результатом віднімання від нього найбільшого елемента праворуч (`arr[i] - maxRight`). Останній елемент = `-1`.
- **Відмінність від основної задачі:** Замість простого запису `maxRight` ми обчислюємо різницю `temp - maxRight`.
- **Підказка щодо модифікації:** Справа наліво: зберігаємо `temp = arr[i]`, якщо це останній елемент — записуємо `arr[i] = -1`, інакше `arr[i] = temp - maxRight`. Оновлюємо `maxRight = Math.max(maxRight, temp)`.
- **Приклад:**
  - Вхід: `arr = [17, 18, 5, 4, 6, 1]`
  - Вихід: `[-1, 12, -1, -2, 5, -1]`.

### 4. [Easy_05_04_ReplaceElements.java](file:///C:/Users/kosti/Projects/google-practice/src/topic01_arrays_strings/practice/Easy_05_04_ReplaceElements.java) — Check If Dominated By Right
- **Мета:** Перевірити, чи кожен елемент (крім останнього) менший або дорівнює найбільшому елементу праворуч від нього.
- **Відмінність від основної задачі:** Повертається булеве значення (`boolean`), сам масив не змінюється.
- **Підказка щодо модифікації:** Обходимо справа наліво. Ініціалізуємо `maxRight = arr[arr.length - 1]`. Йдемо від `arr.length - 2` до `0`. Якщо `arr[i] > maxRight`, негайно повертаємо `false`. Оновлюємо `maxRight = Math.max(maxRight, arr[i])`. Якщо цикл завершено — повертаємо `true`.
- **Приклад:**
  - Вхід: `arr = [3, 4, 5]`
  - Вихід: `true` (3 <= 5, 4 <= 5).

### 5. [Easy_05_05_ReplaceElements.java](file:///C:/Users/kosti/Projects/google-practice/src/topic01_arrays_strings/practice/Easy_05_05_ReplaceElements.java) — Replace with Greatest Inclusive
- **Мета:** Замінити кожен елемент на найбільший елемент праворуч **включно** з самим елементом (без -1 для останнього).
- **Відмінність від основної задачі:** Поточний елемент бере участь у визначенні результату для своєї позиції.
- **Підказка щодо модифікації:** Обхід справа наліво. Оскільки елемент є частиною діапазону, оновлення максимуму відбувається перед записом, або просто: `maxRight = Math.max(maxRight, arr[i])` та `arr[i] = maxRight`. Ініціалізація `maxRight = Integer.MIN_VALUE`.
- **Приклад:**
  - Вхід: `arr = [1, 5, 2]`
  - Вихід: `[5, 5, 2]`.
